What is REST?

REST (Representational State Transfer) is an architectural style that defines a set of constraints and principles for creating web services. These are recommendations, not strict standards, designed to create scalable, maintainable, and flexible APIs.

Core Principles

  • • Resources: Everything is treated as a resource
  • • URIs: Each resource has a unique identifier
  • • Representations: Resources can have multiple formats
  • • Stateless: No client context stored on server
  • • HTTP Methods: Standard operations for resource management

The Six Constraints of REST

1. Client-Server

Separation of concerns between UI/client and data storage/server. This allows:

  • Independent evolution of components
  • Better scalability
  • Improved portability

2. Stateless

Each request contains all information needed to complete it:

  • No session dependency
  • Better reliability
  • Easier scaling

3. Cacheable

Responses must define themselves as cacheable or not:

  • Improved performance
  • Better scalability
  • GET, PUT, DELETE are idempotent

4. Layered System

Client can't tell if connected directly to server:

  • Load balancing
  • Shared caches
  • Security layers

5. Code on Demand (Optional)

Server can extend client functionality:

  • JavaScript execution
  • Dynamic client updates
  • Flexible implementation

6. Uniform Interface

Standardized resource interaction:

  • Resource identification in requests
  • Resource manipulation through representations
  • Self-descriptive messages

HATEOAS

HATEOAS (Hypermedia As The Engine Of Application State) is a constraint that makes REST APIs self-discoverable. Like a "choose your own adventure" book, clients interact with the API dynamically through hypermedia links provided by the server. 

{
  "id": 1,
  "name": "John Doe",
  "_links": {
    "self": { "href": "/users/1" },
    "profile": { "href": "/users/1/profile" },
    "orders": { "href": "/users/1/orders" }
  }
}

In this example, the API response includes links to related resources, allowing the client to discover and navigate the API dynamically.

RESTful Best Practices

Resource Naming

  • Use nouns for resources
  • Use plural forms for collections
  • Use hierarchical relationships
  • Keep URLs simple and readable

HTTP Methods

  • GET for reading
  • POST for creating
  • PUT/PATCH for updating
  • DELETE for removing
